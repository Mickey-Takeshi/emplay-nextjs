# EMPLAY ブログコンテンツ作成・更新マニュアル

> 対象: `emplay.jp` のブログを作成・更新する編集者、ディレクター、実装担当者  
> 本番リポジトリ: `Mickey-Takeshi/emplay-nextjs`  
> 最終更新: 2026-07-15

## 1. このマニュアルの目的

検索順位のためだけに情報量を増やすのではなく、読者が自社で判断・実行できる記事を継続して公開する。Googleにクロールされること、正規URLとして認識されること、内容に独自の価値があることを分けて確認する。

次のような記事は公開しない。

- 見出し、表、箇条書きを並べただけで、判断基準の説明がない
- 製品名や機能を羅列しているが、誰に向くか、どう選ぶかが分からない
- 根拠を確認できない統計、料金、成果数値、架空事例が含まれる
- 公開年をタイトルへ付けたまま、翌年以降の更新担当・更新日が決まっていない
- 既存記事と検索意図が重複し、どちらを正規記事にするか決まっていない

## 2. 公開前に決めること

記事を書き始める前に、以下を1枚にまとめる。

| 項目 | 決める内容 |
|---|---|
| 主軸キーワード | 1記事につき1つ |
| 検索意図 | Know / Do / Compare / Buyのどれか |
| 想定読者 | 業種、役割、知識、直面している課題 |
| 読後の状態 | 読者が判断・実行できるようになること |
| 一次情報 | 公式資料、自社経験、検証結果、顧客から多い質問 |
| 既存記事 | 重複する記事と統合・内部リンクの方針 |
| 更新責任 | 料金・制度・仕様をいつ誰が見直すか |

既存記事と同じ検索意図の場合は、新規記事を増やさず既存記事へ統合する。

公開前には、記事タイトルだけでなくslugと見出し構成も検索する。似た記事がある場合は、次の順で判断する。

1. 読者の課題と読後の行動が同じなら既存記事へ追記する
2. 入門・比較・実践など検索意図が明確に異なる場合だけ別記事にする
3. 統合時は評価を残す正規URLを1つ決め、旧URLを恒久転送する
4. sitemap、カテゴリ一覧、関連記事、本文内リンクから旧URLを除く

## 3. タイトル・slug・概要文

### タイトル

- 主要キーワードを前半へ置く
- 30〜45文字を目安に、内容を過不足なく示す
- 「完全」「徹底」「最新」は、内容がその表現に見合う場合だけ使う
- 年号は原則として付けない
- 年度制度や年次比較など、年が検索意図そのものである場合のみ年号を付ける

良い例:

```text
CMS比較ガイド｜運用体制・費用・拡張性から選ぶ方法
SNS運用の始め方｜媒体選定から効果測定まで解説
```

避ける例:

```text
【2026年最新】おすすめツール完全ガイド
```

更新されなければすぐに古くなり、内容の信頼性も下がる。

### slug

- 英小文字とハイフンを使用する
- 年号は含めない
- 公開後は原則変更しない
- 変更する場合は、旧URLから最も近い現行URLへ恒久リダイレクトを設定する

### excerpt

- 100〜160文字程度
- 記事で分かること、対象読者、得られる判断材料を書く
- タイトルを言い換えただけの文章にしない

## 4. 記事構成

基本構成は以下とする。

```text
導入
  ↓
定義・前提
  ↓
選び方・判断基準
  ↓
実践手順
  ↓
失敗例・注意点
  ↓
効果測定・改善
  ↓
FAQ
  ↓
まとめ
  ↓
関連記事
```

すべての記事へ同じ見出しを機械的に入れる必要はない。検索意図に不要なセクションは省く。

### 導入文

200〜400文字を目安に、次の3点を書く。

1. 読者が抱える状況や判断の難しさ
2. この記事の結論または重要な前提
3. 記事を読むことで分かること

「本記事では詳しく解説します」だけで終わらせない。

### H2直下の説明

各H2の直後には、原則として120〜300文字の平文を置く。

- なぜこの項目が重要か
- 何を基準に判断するか
- 前後のセクションとどうつながるか
- 実務では何に注意するか

表や箇条書きからセクションを始めない。先に読み方や判断軸を説明する。

### 表・箇条書き

表と箇条書きは情報を整理する補助手段であり、本文の代わりではない。

- 表の前に比較軸と使い方を説明する
- 表の後に「この結果からどう選ぶか」を説明する
- 箇条書きが2ブロック続いたら、平文で意味や判断を補う
- 製品カタログ型の記事では、各製品紹介より先に選定基準を書く

## 5. コンテンツ量と解説密度

文字数だけを合格基準にしない。目安は以下とする。

| 指標 | 基準 |
|---|---|
| 本文量 | 通常記事5,000〜8,000字を目安。検索意図を満たせば増減可 |
| 平文比率 | 原則45%以上 |
| カタログ・比較記事 | 平文比率35%以上に加え、選定基準と結論を必須とする |
| H2直下 | 120〜300文字程度の説明 |
| 1段落 | 2〜4文。長すぎる場合は意味のまとまりで分ける |
| 内部リンク | 文脈に合う2〜5本 |
| 外部根拠 | 変更されやすい事実は公式一次情報を1〜3本 |

平文比率は、Markdown記号を除いた本文のうち、見出し・表・箇条書き以外の文章が占める割合とする。

次の場合は、文字数が多くてもリライト対象にする。

- 平文比率が35%未満
- 1段落が短い補足文ばかりで、理由や具体例がない
- 見出し数と表・リスト行数が平文段落数を大きく上回る
- 読者が比較後にどれを選ぶべきか分からない

## 6. 情報の正確性

### 一次情報を優先する

料金、制度、法令、製品仕様、利用者数などは、公式サイト、公的機関、製品ドキュメントを確認する。

- 他社ブログを根拠にしない
- 確認日を編集メモへ残す
- 将来変わる数値は断定せず、公式ページの確認を促す
- 根拠のない「成功事例A社」「○%改善」を作らない

### 制度・法令・セキュリティ記事

補助金、税務、個人情報、セキュリティなどは公開前に別途ファクトチェックを行う。制度が終了・変更している場合は、年号だけを差し替えず、記事の統合、アーカイブ、noindex、リダイレクトを含めて判断する。

助成金・補助金は、サービスページ、会社案内、共通CTAなどの営業訴求には使用しない。一方、ブログで制度を一般的・中立的に解説することは許容する。記事内では申請代行、採択保証、自社サービスの値引き根拠として訴求せず、申請時点の公式情報を確認するよう明記する。

## 7. SEOとインデックス要件

コンテンツ品質と技術的インデックス要件を混同しない。

### 記事ページに必要な状態

- HTTP 200を返す
- `robots` が `index, follow`
- 自己参照canonicalがある
- sitemap.xmlに正規URLだけが載っている
- フッターまたはHTMLサイトマップから2クリック以内で到達できる
- 本文が初期HTMLに含まれる
- H1が1つで、H2→H3の順に並ぶ
- ArticleとBreadcrumbListの構造化データが本文と一致する
- 公開日と更新日が正しい

### Search Consoleで対象外になった場合

以下の順で確認する。

| 確認結果 | 主な理由 | 対応 |
|---|---|---|
| 404 | URL変更、削除、過去ビルド資産 | 代替記事があれば恒久転送。資産URLは原則対応不要 |
| 3xx | 旧URL | 転送先が最も近い内容か確認し、sitemapから旧URLを除外 |
| noindex | 意図的または設定ミス | 公開方針を確認して修正 |
| canonical不一致・未設定 | 重複候補をGoogle側で判定 | 自己参照canonicalと内部リンクを修正 |
| sitemap未掲載 | 発見性不足 | 正規URLをsitemapへ追加し再デプロイ |
| クロール済み未登録 | 品質、重複、鮮度、サイト評価 | 検索意図、独自性、解説密度、内部リンクを見直す |
| 検出・未クロール | クロール優先度、リンク不足 | 内部リンク、sitemap、サーバー状態を確認 |

Search Consoleの「前回のクロール日」だけでは除外理由を断定しない。URL検査画面の理由ラベルと、公開側の技術診断を合わせて判断する。

### リライト後

1. `updated_at`を更新する
2. sitemapの`lastModified`へ`updated_at`を使用する
3. Vercelで再ビルドする
4. 本番HTMLでcanonical、本文、構造化データを確認する
5. Search ConsoleのURL検査で「公開URLをテスト」する
6. 問題がなければインデックス登録をリクエストする

## 8. 内部リンク

- アンカーテキストにはリンク先の主題を書く
- 「こちら」「詳しくはこちら」だけにしない
- ピラー記事と詳細記事を相互リンクする
- 404または転送を挟む旧URLへリンクしない
- 同じリンクを関連記事欄へ大量に並べない

関連記事の形式:

```markdown
## 関連記事

- [記事タイトル](/blog/slug) - この記事との関係
```

## 9. 画像

- サムネイルは原則としてローカル管理のWebPを使用する
- 内容と関係の薄いストック写真を使わない
- 幅・高さまたはアスペクト比を指定する
- 画像の内容を説明するaltを付ける
- 外部画像を使う場合はHTTP 200、利用条件、表示速度を確認する

## 10. 実装方法

記事データはSupabaseの`emplay_hp.blog_posts`で管理する。変更は本番リポジトリのマイグレーションとして残す。

### 新規記事

```sql
INSERT INTO emplay_hp.blog_posts (
  title,
  slug,
  excerpt,
  content,
  thumbnail,
  category,
  published_at,
  updated_at
)
VALUES (
  'タイトル',
  'slug-name',
  '概要文',
  $content$Markdown本文$content$,
  '/images/blog/thumbnail.webp',
  'カテゴリ名',
  now(),
  now()
);
```

### 既存記事

```sql
UPDATE emplay_hp.blog_posts
SET
  content = $content$更新後のMarkdown本文$content$,
  updated_at = now()
WHERE slug = 'target-slug';
```

部分更新では、同じマイグレーションを再実行しても文章が重複しない条件を付ける。

## 11. 運用フロー

### 新規記事

1. 検索意図と既存記事の重複を確認
2. 一次情報と更新責任を決定
3. 構成案を作成
4. 本文を執筆
5. 平文比率と見出し構造を確認
6. 編集・ファクトチェック
7. SQLマイグレーション作成
8. `supabase db push --dry-run`
9. `supabase db push`
10. `npm run build`
11. mainへpushしてVercelを再生成
12. 本番HTMLとSearch Consoleを確認

### 既存記事

1. 除外理由または更新目的を特定
2. 技術要因とコンテンツ要因を分ける
3. 表・箇条書きは残し、判断に必要な解説を追加
4. 古い事実とリンクを更新
5. `updated_at`を更新
6. ビルド・本番確認
7. URL検査とインデックス登録リクエスト

## 12. 公開前チェックリスト

### 内容

- [ ] 検索意図が1つに定まっている
- [ ] 導入で結論と読後の価値が分かる
- [ ] 各H2直下に解説がある
- [ ] 表や箇条書きの前後に読み方・判断がある
- [ ] 平文比率が基準を満たす
- [ ] 公式一次情報を確認した
- [ ] 架空の数値・事例がない
- [ ] 年号タイトルに更新計画がある
- [ ] 内部リンクが正規URLを向いている

### 技術

- [ ] HTTP 200
- [ ] 自己参照canonical
- [ ] `index, follow`
- [ ] sitemap掲載
- [ ] H1が1つ
- [ ] Article・BreadcrumbListが本文と一致
- [ ] `updated_at`が更新されている
- [ ] `npm run build`が成功する
- [ ] 本番HTMLに本文が含まれる

## 13. 定期監査

月1回、全記事を以下の基準で確認する。

```bash
npm run audit:seo
```

このコマンドは、重複slug/title、3,000字未満、内部リンク3本未満、リンク切れ、編集対象外トピックを検査し、問題がある場合は終了コード1を返す。

- 404・リダイレクト・canonical
- sitemapとDBのURL差分
- 年号タイトルと古い料金・制度
- 平文比率35%未満の記事
- 内部リンクの404
- Search Consoleの「クロール済み - インデックス未登録」
- 90日以上表示・クリックがない記事

低品質記事を増やすより、重複記事の統合と既存記事の更新を優先する。
